# Pedagogy: build for application, not recall

The page exists so the user *does something differently* afterward (whether they
watched a video or read an article). Recall is a checkpoint, not the destination.
Design backward from "what will they do on Monday?"

## The loop every page should run

For each main idea, walk the user through:

1. **Orient** — where this sits in the whole, why it matters now.
2. **Understand** — the idea explained plainly and deeply (the mechanism / the
   "why it's true", not just the label).
3. **Encode** — make it stick: a vivid example, analogy, or visual.
4. **Retrieve** — the user recalls it without looking (flashcard, quiz, cloze).
5. **Apply** — the user maps it onto their own life (a concrete action / plan).
6. **Reflect** — the user writes back in their own words or commits to a change.

A section that stops at step 2 or 3 is unfinished.

## The rule: explain + test + apply, for every main point

When you plan a point, write its three forms:

| Point | Explain (understand) | Test (retrieve) | Apply (transfer) |
|---|---|---|---|
| e.g. "Procrastination = instant-gratification monkey wins" | concept card + diagram of the two characters | flashcard: "what wakes the panic monster?" | if-then: "If I notice the monkey at 9pm, then I start a 10-min timer" |

If you can't write an "apply" cell for a point, it probably needs a reflection
prompt instead ("where in your life does this show up?"). Never leave it blank.

## Learning-science principles (use them, don't name-drop them)

- **Active recall** beats re-reading. Prefer a question the user answers over a
  bullet they skim. Hide answers behind a flip/reveal so they try first.
- **Spacing & interleaving**: a flashcard deck the user can revisit (with a
  "review again / got it" sort) beats one-shot exposure. Persist progress.
- **Elaboration**: ask "why?" and "how does this connect to X?" not just "what?".
- **Concrete examples**: every abstract idea gets at least one real example from
  the video, plus a prompt for the user to generate their own.
- **Dual coding**: pair words with a visual (diagram, chart, timeline) when the
  idea is structural or quantitative.
- **Generation effect**: make the user produce something (type a reframe, fill a
  plan) rather than only consume.
- **Implementation intentions**: turn advice into "If [situation], then I will
  [action]." This is the single highest-leverage application format.
- **Self-explanation**: a "say it back in your own words" box cements understanding.
- **Desirable difficulty**: don't pre-reveal answers; a little effort improves
  retention. But keep friction in the *thinking*, never in the UI.

## Depth of explanation

Match the source. For an idea-dense talk or essay, go deep on the mechanism.
Signs you've gone deep enough:

- A reader who never saw the source could explain the idea to someone else.
- You answered the implicit "why is this true / why does this work?".
- You surfaced the non-obvious insight, the caveat, or the failure mode, not
  just the headline.

Write in plain, direct language. Short sentences. No filler, no hype, no em
dashes. Define jargon the first time. Keep the author's or speaker's voice and
key phrasing where it's vivid, but don't pad.

## Adapt to the kind of source

This is the main lever for variety. The source type should decide the page's
**structure and its element set**, so two notes rarely share a skeleton. Read
the type off the source, build the shape it implies, and pick the interactions
that fit. Treat the items below as starting points to depart from, never a script
to fill in (and never copy another note's HTML to "match the format").

- **Lecture / talk** — strong concept cards + a quiz; one big "apply to your
  work/life" worksheet at the end.
- **Tutorial / how-to** — ordered steps with checkboxes, a copyable
  checklist/template, "try it now" prompts; flashcards for the gotchas.
- **Coaching / therapy / self-help** — capture each question and the coach's
  move; reframe flip-cards (belief → truer belief); reflection journal; if-then
  plans. This is where application matters most.
- **Interview / podcast** — organize by theme not chronology; pull-quote the
  best lines with commentary; flashcards on the guest's key claims; "which of
  these will you try?" picker.
- **Debate / two-sided** — a side-by-side comparison table of the arguments; a
  "where do you land?" slider + reasoning box.
- **Documentary / explainer** — a timeline; charts for real figures; a "myth vs
  fact" set; quiz.
- **Sales / pitch / persuasion** — extract the framework honestly; add a
  critical-thinking prompt ("what would have to be true for this to work for
  you?").
- **Essay / long-read (article)** — organize by the argument's structure; pull
  the load-bearing claims into concept cards; annotated quotes with the section
  cited; a "where do you agree or disagree, and why?" reflection.
- **News / analysis (article)** — separate fact from interpretation; a short
  "what happened / why it matters / what to watch" frame; quiz on the specifics.

## Content → component cheat sheet

| When the content is… | Reach for… |
|---|---|
| a definition or model | concept card, glossary, hover-define |
| a process or sequence | numbered steps with checkboxes, timeline |
| a comparison / tradeoff | comparison table, two-column flip |
| real numbers or trends | chart (bar/line/radar) — real data only |
| a belief to shift | reframe flip-card (before → after) |
| a memorable claim | annotated pull-quote (video: timestamp button opens the modal; article: cite the section) |
| something to memorize | Anki-style flashcard deck |
| a check of understanding | MCQ quiz, cloze (fill-the-blank) |
| advice to act on | action checklist, if-then plan, habit tracker |
| a personal connection | reflection journal, "your situation" input |
| a decision to practice | scenario / what-would-you-do prompt |
| a self-rating or diagnostic | self-assessment scorecard, "where do you land?" slider |
| a priority / ranking call | tap-to-order ranking exercise |
| pairs to connect (term ↔ meaning) | matching exercise |
| a misconception to bust | myth-vs-fact toggle |
| a multi-step method to rehearse | interactive step-runner, branching decision tree |

Don't try to use every row on one page. Pick the handful the source actually
needs, and let the choice differ from your last note.

## Anti-patterns

- A wall of summary bullets with nothing to do. (The thing the user disliked.)
- Reusing the same skeleton and widget set on every source (deck + quiz +
  if-then + reflection no matter the content). Let the source type vary both.
- Copying an earlier note's HTML to "match the format." There is no format.
- Tests with no application, or application with no understanding.
- Fabricated stats, quotes, studies, or "the source said" claims that aren't in
  the source. If unsure, leave it out or mark it as your own framing.
- Quizzes whose answers are obvious or whose wrong options are silly.
